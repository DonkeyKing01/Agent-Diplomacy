## 1. 对局对象

### 国家

后端固定 10 个国家：

| id | 名称 | 简称 |
|---|---|---|
| `aur` | 一排领地 | 一排 |
| `mar` | 二排领地 | 二排 |
| `vel` | 三排领地 | 三排 |
| `kaz` | 四排领地 | 四排 |
| `sol` | 五排领地 | 五排 |
| `nor` | 六排领地 | 六排 |
| `ferr` | 七排领地 | 七排 |
| `zeph` | 八排领地 | 八排 |
| `dra` | 九排领地 | 九排 |
| `ith` | 十排领地 | 十排 |

### 地图格类型

每个地块有：

- `land`：陆地。
- `coast`：海岸，可被陆军和舰队进入。
- `sea`：海域，只能舰队进入。
- `mountain`：代码里保留类型判断，但当前地图主要使用 `land/coast/sea`。

补给中心由 `sc: true` 标记。只有 `land` 和 `coast` 会计入所有权；`sea` 不会被国家拥有。

### 邻接

地图是六边形网格。每个地块的邻接关系由六个 hex 方向自动计算：

```text
(1,0), +(1,-1), +(0,-1), +(-1,0), +(-1,1), +(0,1)
```

所有移动、支援、撤退的基础合法性都依赖这个邻接关系。

## 2. 年度阶段

每年固定 6 个阶段：

| phase key | 中文阶段 | 是否下军令 |
|---|---|---|
| `spring` | 春季·谈判与决策 | 是 |
| `springRetreat` | 春季撤退 | 否 |
| `autumn` | 秋季·谈判与决策 | 是 |
| `autumnRetreat` | 秋季撤退 | 否 |
| `winter` | 冬季调整 | 否 |
| `review` | 年度复盘与治理 | 否 |

阶段推进顺序固定。`review` 之后进入下一年的 `spring`。

## 3. 初始化与开局

### 初始化

调用 `/api/v1/game/init` 后：

- 年份设为 `1901`。
- 阶段设为 `spring`，但状态是 `preparing`。
- 所有本土补给中心归属各自国家。
- 初始单位按后端 `INITIAL_UNITS` 固定生成。
- 每个国家生成默认 `system_prompt`、`skills_md`、`memory`、`annual_advice`。
- 清空或重置密信、战报、年鉴、智能体配置。

### 开局前强制条件

调用 `/api/v1/game/start` 离开准备阶段前，所有国家都必须有非空：

- `system_prompt`
- `skills_md`
- `annual_advice`

缺任何一个都会被拒绝开局。

## 4. 单位与地形

### 陆军

陆军可进入：

- `land`
- `coast`

陆军不能直接进入 `sea`。

### 舰队

舰队可进入：

- `coast`
- `sea`

舰队不能进入 `land`。

### 所有权与单位位置

单位位置和地块所有权是两个概念：

- 单位进入一个非海域地块后，该地块所有权变为该单位国家。
- 海域不会产生所有权。
- 没有单位的地块仍然保留之前的所有权，直到被别国单位进入。

## 5. 军令类型

后端接受四种命令：

| action | 含义 |
|---|---|
| `Move` | 移动/进攻 |
| `Hold` | 原地防守 |
| `Support` | 支援 |
| `Convoy` | 舰队护送陆军跨海 |

每支单位最多只接受一条命令。重复提交同一单位命令时，只保留第一条。

没有合法命令的单位，在裁定器中默认视为 `Hold`。

## 6. LLM 军令校验

正式推进时，LLM 产出的命令先经过 `_validate_orders` 清洗。

### Move 校验

`Move` 必须满足：

- 起点必须是自己的单位所在地。
- 目标必须是相邻地块。
- 陆军不能进海。
- 舰队不能进纯陆地。

不满足则改为 `Hold`。

### Support 校验

`Support` 必须满足：

- `support_of` 必须是当前棋盘上一支单位的位置。
- `target` 必须是存在的地块。
- 支援单位自己必须能移动到 `target`。
- 如果是支援防守，即 `target == support_of`，支援单位还必须与被支援单位相邻。

不满足则改为 `Hold`。

### Convoy 校验

`Convoy` 必须满足：

- 执行单位必须是舰队。
- 执行单位必须位于 `sea`。
- `support_of` 必须是一支陆军所在位置。
- `target` 必须是 `coast`。

不满足则改为 `Hold`。

### 主动补位规则

校验完成后，后端会对空缺命令或 `Hold` 做主动补位：

- 如果单位有明显收益较高的合法移动，后端会把 `Hold` 升级为 `Move`。
- 评分大致优先级：
  - 相邻中立补给中心最高。
  - 相邻敌方补给中心次之。
  - 中立非补给中心。
  - 敌方非补给中心。
  - 已方地块会扣分。
- 同一国家的补位移动会避免重复认领同一个目标。

这意味着模型想原地不动时，后端可能仍会让它主动扩张。

## 7. 军事裁定

裁定入口是 `adjudicate(...)`，只在 `spring` 和 `autumn` 决策阶段执行。

### 基础强度

- 每个移动进攻默认强度为 `1`。
- 每个原地防守默认强度为 `1`。
- 每个有效支援提供 `+1`。

### 谁算防守者

只有“没有移动出去”的单位才算防守者。

如果某单位从 A 移动到 B：

- 它不再防守 A。
- 即使移动失败，也不会在 A 的战斗中提供防守强度。

地块所有权本身不提供防守。必须有单位原地防守才有防守强度。

### 同国重复进攻

当前规则已经修正为：

- 同一国家多个单位攻击同一目标时，不会互相对冲。
- 也不会把多个单位自动叠加成更高强度。
- 后端只保留该国家对该目标“强度最高的一路”参与裁定。
- 如果同国多路强度相同，则按订单读取顺序保留一路。

例：一排两个强度 1 的单位同时打九排高地，不再判为一排自撞；只按一支强度 1 的一排进攻参与比较。

### 多国同强度进攻

如果多个国家对同一目标的最高强度相同，且没有防守者并列最高：

- 判为 `争夺`。
- 没有任何一方进入。
- 地块所有权不变。
- 该地块记为 standoff，撤退时不能退入。

### 防守者并列最高

如果防守者在最高强度中：

- 防守者胜。
- 判为 `防守`。
- 所有进攻失败。

因此，攻击一个有单位防守的地块时，进攻强度必须严格大于防守强度。

### 无防守者时的占领

如果目标地没有原地防守单位，且存在唯一最高强度进攻者：

- 该进攻者进入目标地。
- 非海域目标的所有权改为进攻者。
- 战报记为 `占领变化`。

### 有防守者时的进攻

如果目标地有原地防守单位，且进攻者强度严格高于防守者：

- 进攻者进入目标地。
- 防守单位被击退。
- 非海域目标的所有权改为进攻者。
- 战报记为 `进攻`。
- 被击退单位进入撤退列表。

### 禁止攻击自己原地单位

如果目标地有本国单位，且该本国单位正在原地防守，则本国其他单位不能把它“顶走”。

结果：

- 判为 `争夺`。
- 进攻失败。

### 直接换位

当前裁定是按目标地分别处理，没有完整实现经典 Diplomacy 的所有交换/头撞细则。

因此 A->B、B->A 的表现取决于两个目标地各自的局部裁定：

- 如果双方都移动出去，原地都不算防守。
- 两个目标格可能分别被对方占领，形成所有权互换。
- 如果其中一边还被第三方影响，可能出现一边成功、一边失败，甚至看起来像“双吃”。

### 被占原地与移动失败

单位移动出去后，它的原地如果被别人成功占领：

- 该单位不会继续作为原地防守者。
- 如果它自己的移动失败，且原地已被占用，后续重建单位列表时它可能无法回到原地。

这是当前实现的简化点。

## 8. 支援规则

### 支援进攻

支援进攻需要：

- 支援单位执行 `Support`。
- `support_of` 是被支援单位的起点。
- `target` 是被支援单位实际移动的目标。
- 支援单位自己能直接移动到 `target`。

有效后，被支援进攻强度 `+1`。

### 支援防守

支援防守写法：

- `support_of == target`

要求：

- 被支援位置有单位。
- 支援单位与该位置相邻。
- 支援单位能进入该地形。

有效后，被支援防守强度 `+1`。

### 支援可跨国家

后端允许支援外国单位。

这类行为还会被外交系统识别为合作或停火事件。

### 支援被切断

如果支援单位所在格被攻击，支援可能被切断。

切断条件：

- 有敌方单位攻击支援单位所在地。
- 攻击者不是被支援行动的相关豁免来源。
- 同国攻击不会切断本国支援。

被切断后，该支援不提供 `+1`。

## 9. 护送规则

裁定器本身支持多海域 Convoy：

- 只有海域中的舰队能执行 `Convoy`。
- `support_of` 必须是一支陆军起点。
- `target` 必须是海岸。
- 起点旁边必须有参与护送的海域舰队。
- 目标海岸旁边必须有参与护送的海域舰队。
- 参与护送的舰队之间必须能通过连续相邻海域连通。

但正式 LLM 推进路径里还有一个限制：

- `_validate_orders` 当前要求 `Move` 目标必须相邻。
- 所以非相邻的陆军跨海移动，即使裁定器支持，也可能在进入裁定前被校验成 `Hold`。

换句话说：裁定器能力支持远程护送，但正式 LLM 军令校验目前主要支持相邻移动。

## 10. 撤退规则

被击退单位会生成 `pending_retreats`。

每个撤退项包含：

- `owner`
- `type`
- `location`
- `attacked_from`
- `legal_retreats`

合法撤退地必须满足：

- 与被击退位置相邻。
- 不能是攻击来源地。
- 不能是当前已有单位占据的地。
- 不能是本阶段发生 `争夺` 的地。
- 单位类型必须能进入该地形。

撤退阶段由 LLM 选择：

- `RETREAT` 到合法撤退地。
- 或 `DISBAND` 解散。

如果 LLM 给出的撤退目标不合法，后端按解散处理。

撤退到非海域地块时，该地块所有权会变为撤退单位国家。

## 11. 冬季调整

冬季根据补给中心数和单位数调整。

### 目标单位数

每个国家的目标单位数：

```text
目标单位数 = 当前拥有的补给中心数
```

### 增兵

如果补给中心数大于单位数：

- 可增兵数量为差值。
- 只能在本国 `HOME_CENTERS` 中增兵。
- 该本土中心必须仍由本国拥有。
- 该格不能已有单位。
- LLM 可提出 `builds`。
- 非法 build 会被跳过。
- 如果 LLM 少提或提非法，后端会在可用本土中心自动补足。
- 海岸可建舰队或陆军；如果在非海岸提舰队，会改成陆军。
- 自动补足时，海岸默认建舰队，非海岸建陆军。

### 裁军

如果补给中心数小于单位数：

- 必须裁掉差值数量。
- LLM 可提出 `disbands`。
- 非法 disband 会被跳过。
- 如果 LLM 少提，后端按当前单位列表顺序补足裁撤。

### 无调整

如果补给中心数等于单位数：

- 不增兵。
- 不裁军。

## 12. 灭国规则

每次推进阶段末尾都会检查补给中心数。

如果某国家补给中心数 `<= 0`：

- 该国家加入 `eliminated_nations`。
- 该国家所有单位从棋盘移除。
- 之后该国家不再参与谈判、决策、冬季调整。
- 该国家智能体配置不能再被编辑。

## 13. 外交与密信

### 谈判轮数

每个春季/秋季决策阶段：

1. 先进行 2 轮谈判。
2. 每轮每个未灭国国家最多发 0 到 2 条消息。
3. 再进行一次最终军令决策。
4. 最终决策也可附带 0 到 2 条消息。

消息会写入数据库，并进入后续信任和记忆计算。

### 可联系对象

某国家可联系对象由距离决定：

- 如果双方势力存在位置距离 `<= 3`，可联系。
- 如果距离更远，只在偶数 tick 可联系。
- `public` 永远可作为公开频道。

tick 计算：

```text
tick = (year - 1901) * 6 + phase_index
```

### 消息校验

每次模型输出最多保留前 2 条消息。

规则：

- 空消息丢弃。
- 目标不是允许对象时，改发 `public`。
- 不能发给自己；发给自己会改成 `public`。
- 完全重复的 `(from, to, content)` 会去重。

## 14. 信任与软联盟

后端为每个有方向关系维护一个信任分，初始为 `50`。

### 消息影响

消息内容通过关键词计算信号：

- 正向词，例如合作、互助、支援、结盟、停火等。
- 负向词，例如威胁、进攻、背叛、欺骗等。
- 承诺词，例如互不侵犯、不攻击、共同进攻、停火等。

粗略公式：

```text
score = min(正向词数 * 4 + 承诺词数 * 3, 16) - min(负向词数 * 6, 18)
```

私信影响接收者对发送者的信任。

公开消息影响所有其他国家对发送者的信任。

信任分限制在 `0-100`。

### 战报外交事件影响

战报中的外交事件会继续修正信任：

- `betrayal`：
  - 受害者对行动者 `-22`
  - 行动者对受害者 `-8`
- `cooperation`：
  - 目标对行动者 `+10`
  - 行动者对目标 `+6`
- `truce`：
  - 双方各 `+4`

### 软联盟等级

软联盟由信任分和历史互动计算：

| 等级 | 条件概要 |
|---|---|
| `hostile` | 被背刺过，或信任 `<= 35` |
| `allied` | 信任 `>= 72`，且有承诺或协同作战 |
| `coordination` | 信任 `>= 60`，且有承诺或停火 |
| `non_aggression` | 信任 `>= 52` |
| `neutral` | 其他情况 |

软联盟只是给 LLM 的上下文，不是硬规则。后端允许背叛。

## 15. 外交事件识别

阶段战报中会识别三类外交事件：

### betrayal

如果某国家 `Move` 攻击另一个国家拥有或驻军的地块，并且行动者对受害者的软联盟等级是：

- `non_aggression`
- `coordination`
- `allied`

则记录为 `betrayal`。

### cooperation

如果某国家 `Support` 支援外国单位，且双方软联盟等级是：

- `non_aggression`
- `coordination`
- `allied`

则记录为 `cooperation`。

### truce

如果某国家 `Support` 支援外国单位，但双方还不是软联盟，则记录为 `truce`。

## 16. 记忆系统

后端会维护每个国家的持久记忆 `memory`。

记忆内容包括：

- 最近更新时间。
- 信誉白名单。
- 血仇黑名单。
- 历史偏见。
- 近期局势摘记。

记忆基于：

- 历史密信。
- 战报。
- 背叛、合作、停火事件。
- 信任分。

`memory` 不能被前端或 API 直接人工编辑。每次阶段推进后，后端会自动刷新。

## 17. 年度复盘

`review` 阶段推进时：

- 不下军令。
- 生成战报。
- 调用 LLM 生成年度总结。
- 记录当前补给中心快照。
- 写入 `chronicles`。

如果当前年份达到 `max_year`，则对局状态变为 `finished`。

默认 `max_year = 1910`，可以在准备阶段通过配置修改，但不能低于 `1901`。

## 18. 治理规则

### 可编辑阶段

智能体配置只允许在：

- `preparing`
- `review`

阶段编辑。

其他阶段编辑会被拒绝。

### 可编辑字段

允许编辑：

- `system_prompt`
- `skills_md`
- `annual_advice`

`memory` 不能直接编辑。

### 开局准备阶段

在 `preparing`：

- `system_prompt` 可写。
- `skills_md` 可写。
- `annual_advice` 可写。
- 不计入后续“每国一次修改机会”。
- 年度建议立即对当前年生效。

### 开局后

在 `review`：

- 每个国家的 `system_prompt` 只能修改一次。
- 每个国家的 `skills_md` 只能修改一次。
- `annual_advice` 每年可写，并记录到该国家对应年份。
- 年度建议在非准备阶段写入时，从下一年开始生效。

例：1902 年度复盘阶段写入年度建议，实际在 1903 年及之后作为 active advice 输入给 LLM。

### 年度建议强制

开始游戏前，所有国家必须有年度建议。

年度复盘阶段，后端有函数用于检查某年哪些未灭国国家缺年度建议；当前具体强制点主要体现在开局 `/start`，以及模板同步/治理状态记录。

## 19. 本地 Markdown 模板同步

接口：`/api/v1/game/local_templates/sync`

模板目录：

```text
docs/governance_templates/
```

结构：

```text
preparing/system_prompt/
preparing/skills/
yearly_advice/<year>/
```

文件名固定：

```text
01_第一排.md
02_第二排.md
...
10_第十排.md
```

### 准备阶段同步

`preparing` 时同步：

- `system_prompt`
- `skills_md`
- 当前年份 `annual_advice`

空文件跳过。

内容与数据库一致时跳过。

非空且变化的文件会真实写入数据库中的国家智能体配置。

### 年度复盘同步

`review` 时同步：

- 下一年目录下的 `annual_advice`

例如 1902 年 `review` 同步的是：

```text
docs/governance_templates/yearly_advice/1903/
```

`system_prompt` 和 `skills_md` 在 review 阶段不会通过本地模板同步读取；需要走智能体更新接口。

### 支持年份

模板年份范围：

```text
1901-1915
```

超出范围时年度建议跳过。

## 20. 补给中心手动调整

接口：`/api/v1/game/sc_endowment`

可以直接修改 `sc_json` 中每个国家的补给中心计数。

注意：

- 这只改补给中心计数快照。
- 不直接改地块所有权。
- 后续冬季调整和灭国检查会使用这个计数。

## 21. 战报与历史快照

每次推进都会生成一条战报。

战报包括：

- 阶段。
- 移动命令摘要。
- 外交事件。
- 冲突结果。
- 待撤退列表。
- 冬季调整日志。
- 补给中心排名。
- 灭国信息。

战略地图的历史快照不是直接存完整快照表，而是通过 game log 和战报重新回放生成。

回放时会重新调用裁定器，因此裁定规则修改后，历史快照的回放表现可能与旧战报文字存在差异。

## 22. LLM 调用规则

后端支持三类 provider：

- OpenAI-compatible
- Anthropic
- Gemini

所有 LLM 调用都要求返回 JSON 对象。后端会从代码块或文本中抽取第一个 JSON object。

OpenAI-compatible 调用：

- temperature `0.2`
- JSON object response format
- 对部分兼容模型传 `thinking: disabled`
- 会按配置模型、默认模型、fallback 模型顺序尝试

如果 LLM 决策管线失败：

- 阶段推进失败。
- 返回 `502`。
- 不会使用 `fallback_orders` 自动继续推进。

`fallback_orders` 和 `fallback_messages` 目前是后端工具函数，但正式 `/advance` 路径没有在 LLM 失败时启用它们。

## 23. 当前已知实现简化

这些不是设计建议，而是当前代码事实：

1. 所有权本身不防守，只有没移动的单位防守。
2. 移动出去的单位不防守原地，即使移动失败也一样。
3. 直接换位、对冲、三方进入等复杂情况按目标格独立裁定，没有完整复刻经典 Diplomacy 全部细则。
4. 同国多路攻击同一格已不自撞，但也不自动叠加强度。
5. 裁定器支持跨海护送，但正式 LLM 军令校验目前会拦掉非相邻 `Move`。
6. 冬季自动补建和补裁会在 LLM 提案不足时兜底。
7. `Hold` 可能被后端主动补位改成扩张移动。
8. 灭国后国家不可编辑，也不再行动。
9. 记忆不可人工直接编辑，只能后端自动刷新。
10. 已生成战报是文本记录；历史地图快照是回放生成，规则变更可能影响回放视觉结果。
